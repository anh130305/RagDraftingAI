## MỤC TIÊU DỰ ÁN
Xây dựng hệ thống RAG (Retrieval-Augmented Generation) để hỗ trợ soạn
thảo văn bản hành chính Việt Nam, gồm 3 chức năng chính:
  1. Truy xuất điều khoản pháp luật liên quan (legal retrieval)
  2. Gợi ý biểu mẫu hành chính phù hợp (form retrieval)
  3. Few-shot retrieval ví dụ văn bản đã soạn sẵn (example retrieval)

4. VECTOR DATABASE — embedAndIndex.ipynb  ✅ HOÀN THÀNH
================================================================

Chạy trên: Kaggle Notebook, GPU Tesla T4
Input : dataset/chunks/*.parquet  (upload lên Kaggle dataset "rag-chunks")
Output: chromadb_output.zip  (tải về, giải nén thành thư mục dataset/chromadb/)

⚠️ Cần RE-EMBED sau khi filter — upload lại legal_chunks.parquet mới lên Kaggle
   trước khi chạy embedAndIndex.ipynb

Embedding model: intfloat/multilingual-e5-large-instruct
  - Dim       : 1024
  - Max seq   : 512 tokens
  - Asymmetric retrieval:
      Index (passage): prefix "passage: {text}"
      Query (search) : "Instruct: {task}\nQuery: {query}"
  - Distance  : cosine  (embeddings đã L2-normalize)
  - Batch size: 128 (T4 GPU)

ChromaDB Collections (sau filter):
  legal_chunks    → ~165,000 chunks  (~2,600 MB trên disk)  ← giảm từ 9,400 MB
  forms_chunks    → 10 chunks
  examples_chunks → 150 chunks
  Tổng: ~165,160 chunks | DB size: ~2.6 GB  ← giảm từ ~9.4 GB

⚠️ QUAN TRỌNG — ChromaDB ID vs Parquet chunk_id:
  ChromaDB lưu mỗi chunk với ID = SHA-256[:24] của:
    f"{doc_id}|{chunk_index}|{text[:200]}"
  → ID này KHÁC HOÀN TOÀN với chunk_id text-key trong parquet
  → Metadata phải lấy trực tiếp từ ChromaDB query response (include=["metadatas"])

Retrieval instructions (asymmetric):
  legal   : "Instruct: Tim dieu khoan phap luat Viet Nam lien quan.\nQuery: "
  forms   : "Instruct: Tim mau bieu hanh chinh phu hop voi nhu cau soan thao.\nQuery: "
  examples: "Instruct: Tim vi du van ban hanh chinh tuong tu tinh huong sau.\nQuery: "

6. HYBRID RETRIEVAL — hybridRetrieval.ipynb  ✅ HOÀN THÀNH (đã fix bugs)
================================================================

Pipeline:
  Query → [BM25 top-50] ─┐
                          ├─ RRF (k=60) → top-20 → Cross-Encoder → top-k
  Query → [Dense top-50] ┘

Components:
  BM25     : rank_bm25.BM25Okapi | normalize không dấu, persist pkl
  Dense    : intfloat/multilingual-e5-large-instruct (asymmetric)
  Fusion   : Reciprocal Rank Fusion (k=60)
  Reranker : BAAI/bge-reranker-v2-m3 (multilingual, max_length=512)
  Device   : CPU local (chạy GPU nếu có)

Tham số:
  BM25_TOP_K   = 50   DENSE_TOP_K  = 50
  RRF_K        = 60   RERANK_TOP_N = 20   FINAL_TOP_K = 5

Lưu ý hiệu năng (CPU 16GB RAM):
  - Mỗi query hybrid mất ~18–26s trên CPU (chủ yếu do reranker + embed model)
  - Embed model (multilingual-e5-large) thiết kế cho GPU — chậm trên CPU
  - Không nên đổi embed model (sẽ phải re-embed ChromaDB)
  - Có thể tắt reranker khi dev: use_reranker=False → ~2–3s/query

BM25 cache (sau filter):
  dataset/bm25/bm25_legal.pkl    (~160 MB, giảm từ ~400 MB)
  dataset/bm25/bm25_forms.pkl
  dataset/bm25/bm25_examples.pkl

Bugs đã fix:
  1. _dense_search() thêm "metadatas" vào include=[] → metadata không còn rỗng
  2. meta_lookup (parquet) chỉ làm fallback cho BM25-only hits
  3. CrossEncoder(cache_folder=...) — fix deprecation (cache_dir → cache_folder)
  4. get_embedding_dimension() — fix FutureWarning

Public API:
  retrieve_legal(query, top_k=5, where=None, use_reranker=True)
  retrieve_forms(query, top_k=3, use_reranker=True)
  retrieve_examples(query, top_k=3, form_id=None, use_reranker=True)
  retrieve_all(query, legal_top_k=5, forms_top_k=3, examples_top_k=3,
               legal_filter=None)
    → {"legal": [...], "forms": [...], "examples": [...]}

Filter hợp lệ sau khi giảm dataset:
  type_normalized: "LUẬT" | "NGHỊ ĐỊNH" | "PHÁP LỆNH" | "NGHỊ QUYẾT"
  (THÔNG TƯ và QUYẾT ĐỊNH đã bị loại khỏi dataset)
  ⚠️  Cell 11 (filter test) có test case where={"type_normalized": "THÔNG TƯ"}
      sẽ trả về 0 kết quả — đổi thành "NGHỊ ĐỊNH" hoặc "NGHỊ QUYẾT" khi test.

10. BƯỚC TIẾP THEO
================================================================

  □ RAG Generation Pipeline (ragGeneration.ipynb):
      - Input: user_query (mô tả nhu cầu soạn thảo)
      - Gọi retrieve_all(query) → legal + forms + examples context
      - Ghép context theo template prompt
      - Gọi LLM API → sinh văn bản hành chính
      - Xử lý output: điền placeholder, format markdown

  □ Chọn LLM generation model:
      - GPT-4o / GPT-4o-mini (OpenAI API) — khuyến nghị trước
      - Gemini 1.5 Pro (Google API)
      - Local: Qwen2.5 / Vistral (nếu cần offline)

  □ Prompt templates (promptTemplates.py):
      - System prompt chung (vai trò chuyên viên hành chính VN)
      - User prompt theo từng loại văn bản (công văn, quyết định, nghị quyết...)
      - Format context block: legal_context + form_template + example_context
      - Hướng dẫn điền placeholder {{FIELD}} cho LLM

  □ Evaluation:
      - Retrieval: MRR@5, Recall@5 trên tập test query thủ công
      - Generation: ROUGE, BERTScore, đánh giá thủ công format & nội dung

  □ (Tuỳ chọn) API / UI:
      - FastAPI endpoint: POST /generate {query, form_type?, filters?}
      - Gradio UI cho demo
